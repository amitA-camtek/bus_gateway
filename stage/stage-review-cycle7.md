## Why this cycle found so much after a "READY" verdict

Cycles 5‚Äì6 declared the set DESIGN READY, but **doc 07 (`07-toolconnect-design.md`) was authored *after* those cycles** and received only **one** review pass ‚Äî the CON6 consistency lint. It had never been examined for concurrency, connectivity, security, data-integrity, operations, or test-strategy. This cycle pointed all nine lenses at it (and re-grounded the gateway claims in the shipped `ToolGateway` code). The result: **the gateway's WAL state machine ‚Äî the R-3/R-4 centerpiece ‚Äî is specified as *states* but never as *ownership, atomicity, in-flight, or durability-ordering contracts***, and every "duplicates never leave the gateway" guarantee rests on exactly those unstated contracts. The bus core (docs 06 ¬ß6.2‚Äì6.7, cycles 1‚Äì6) held up well; the failure is concentrated at the gateway edge and the two fab-facing seams (GEM shim, compensation table).

**Honest correction to the record:** the cycles-1‚Äì6 verdict "the design is READY, every fork decided" was true for the bus core but **overstated for the gateway and the GEM seam**, because the artifacts that carry those (doc 07, the transition ring, the GEM accept path) were either not yet written or not yet reviewed at the depth the bus core got. ## The 8 distinct CRITICAL defects (owner-visible; resolutions below are the decided fixes)

### X7-1 ‚Äî WAL entry has no mutation owner and no in-flight state ‚Üí drain/live double-send in steady state, lost `Done` updates
**From:** CC7-1, CC7-2 ¬∑ **Severity: CRITICAL**
`SinkState = {Pending, Done, Poison}` has no *InFlight* state, and ¬ß7.8 asserts `MarkSinkDone` is "thread-safe" with no mechanism. Concurrent actors on one entry-file (Fleet callback, TSMC callback, DrainScheduler, delete predicate) ‚Üí (a) a drain tick returns a `Pending` leg that a live send is *currently* transmitting ‚Üí **duplicate to Fleet/TSMC with no crash**; (b) two completion callbacks read-modify-write the whole entry record ‚Üí one `Done` is lost ‚Üí drain re-sends. Downstream `messageId` dedup was demoted to *optional* (R-3), so nothing catches it. Falsifies "duplicates never leave the gateway."
**Resolution (decided):** (1) add an `InFlight(owner, leaseUtc)` per-leg state; a leg is dispatched only via an atomic `Pending ‚Üí InFlight` transition; `NextPending` never returns `InFlight`; completion `InFlight ‚Üí Done`, failure `InFlight ‚Üí Pending/Poison`; a lease timeout (> sink deadline) reclaims a leg orphaned by a crashed send. (2) **All WAL-entry mutations (mark-done, attempt-increment, poison, delete, quota-gauge) are posted to a single WAL-state actor** ‚Äî one thread owning the entry files and the in-memory index; sink callbacks and the drain post messages, never touch files. Delete-vs-mark and poison-vs-done serialize by construction; a mark for an already-deleted entry is a counted no-op. ‚Üí 07 ¬ß7.4 (states + dispatch-authority rule), ¬ß7.8 (ownership statement replaces "thread-safe"), ¬ß7.3 (`SinkState` gains `InFlight`; `WalEntry` attempts move per-leg).

### X7-2 ‚Äî Dedup identity is specified two incompatible ways; doc 07's per-source variant re-creates the R-2 silent-wafer-loss
**From:** CON7-1, CC7-3, CC7-8a ¬∑ **Severity: CRITICAL**
Four-part key `(source, epoch, topic, seq)` in 01 ¬ß1.4 / 06 ¬ß6.5-dispatcher / 05 A-5, but **three-part `(source, epoch, seq)` per-source** in 06:227 / 07 ¬ß7.2/¬ß7.3/¬ß7.5 / 05 A-6, and `seq` is called "per-(source,epoch)" (06:65) yet used per-topic (06:111). Gateway lanes are per-`(source,topic)` and unordered across topics ‚Üí building `DedupIndex` per ¬ß7.3 verbatim, `tool.telemetry seq=105` lands before `scan.committed seq=100` ‚Üí **fresh wafer dropped as duplicate** = the R-2 failure, reintroduced by doc 07.
**Resolution (decided):** `seq` is **per-`(source, epoch, topic)`** (one sentence in 06 ¬ß6.3); the dedup key is **`(source, epoch, topic, seq)`** everywhere; `DedupIndex` is keyed `(source, epoch, topic)`. ‚Üí fix 06:65/227, 07 ¬ß7.2/¬ß7.3-diagram/¬ß7.5, 05 A-6, snippet 01, snippet 12 header.

### X7-3 ‚Äî Dedup high-water is not durable across entry deletion ‚Üí post-crash redelivery re-appends and re-sends
**From:** CC7-3 ¬∑ **Severity: CRITICAL**
07 ¬ß7.5 says high-water "persisted with the WAL"; 06 ¬ß6.5 says "recovered from the WAL on start." The recover-by-scan variant is unsound once ¬ß7.4 deletion runs: seq=100 arrives ‚Üí routed ‚Üí both sinks Done ‚Üí **entry deleted**; gateway crashes; recovery scans surviving WAL entries ‚Üí high-water = 99; publisher (crashed pre-tombstone) redelivers seq=100 ‚Üí `IsDuplicate=false` ‚Üí **second submission to Fleet+TSMC**. Exactly the duplicate R-3 says is "eliminated to WAL-recovery granularity."
**Resolution (decided):** the high-water is its **own durable per-`(source,epoch,topic)` control record** (small control file / WAL control entries), never derived solely from live entries; ordering contract **`append entry (fsync) ‚Üí persist high-water ‚Üí return handler (=ACK)`**; recovery high-water = `max(control record, max surviving-entry seq)`; deletion never lowers it. ‚Üí reconcile 07 ¬ß7.5 + 06 ¬ß6.5 to one mechanism.

### X7-4 ‚Äî Poison-vs-outage discriminator (`ISink.IsConnected`) is undefined in the design and wrong in both directions in the shipped code
**From:** CN7-1, CC7-4 ¬∑ **Severity: CRITICAL**
¬ß7.4 rule 2 keys the poison/outage split on `IsConnected`, defined nowhere. Verified: `FleetMainServerClientImpl.cs` `IsConnected` is a trailing `volatile bool` ‚Äî true after the last success, false after any `RpcException`. ‚Üí (a) *poison as outage:* one `InvalidArgument` flips the flag false ‚Üí "failed while DISCONNECTED" ‚Üí retry forever = **LB1 resurrected, blessed by the state machine**; (b) *outage as poison:* half-open Fleet, small events keep succeeding (flag true) while one large wafer hits the deadline N times "while CONNECTED" ‚Üí **fresh customer data dead-lettered during a transient slowdown**. `TsmcSink` returns the same `false` for malformed JSON and network failure ‚Äî the bool erases the distinction ¬ß7.4 needs.
**Resolution (decided):** replace the connectivity-flag discriminator with a **per-sink failure taxonomy** the sink returns/throws: `TransientUnavailable` (gRPC UNAVAILABLE/DEADLINE-before-send, socket/init) ‚Üí outage, not an attempt; `DeterministicReject` (INVALID_ARGUMENT / serialization / 4xx-equivalent) ‚Üí poison attempt; `AmbiguousOutcome` (deadline *after* send) ‚Üí leg stays Pending, resend carries `messageId` (X7-5). If `IsConnected` is kept for drain gating it is an **active probe** (channel state), never the last-call flag, with a stated sampling moment. ‚Üí 07 ¬ß7.4 rule 2, ¬ß7.5, ¬ß7.9 (+ tests: "poison while flag stale-false", "deadline storm while healthy small traffic continues").

### X7-5 ‚Äî No P1a dual-run mode in doc 07 ‚Üí the first funded wave double-submits every wafer, or the WAL silently fills and backpressures AOI in normal production
**From:** CN7-2 ¬∑ **Severity: CRITICAL**
03/05 describe a P1a shadowed dual-run (BusSource beside :5005), but doc 07 ‚Äî the normative gateway spec ‚Äî has **zero** dual-run content: ¬ß7.2 wires BusSource‚ÜíWAL‚Üísinks unconditionally. The :5005 leg (`EventMessage`, no envelope) can't match the gateway dedup key. If BusSource routes to sinks at P1a ‚Üí **double Fleet/TSMC submission for the whole retention window** (messageId dedup is optional). If it doesn't ‚Üí WAL entries appended + ACK'd but never completed ‚Üí WAL ‚Üí quota ‚Üí **withhold-ACK backpressure into AOI's journal during normal production, no outage**.
**Resolution (decided):** add "¬ß7.x Dual-run (P1a)" to doc 07 ‚Äî exactly one door routes to sinks (signed-profile flag); the shadow door terminates at a **gateway-side comparator tap** (pair key `correlationId`/`UniqueId`), never at EventRouter; shadow WAL entries are marked `ShadowDone` on comparator consumption (quota-exempt audit class); the flip procedure names the old-spool/WAL overlap arbitration (UniqueId-keyed suppression for N minutes). ‚Üí cross-ref 03 P1a edge + 05 ¬ß5.2.

### X7-6 ‚Äî CMM proxy has authentication but NO per-operation authorization; certs are not operation-scoped ‚Üí any accepted :5007 cert reaches the full :50055 surface
**From:** SEC7-1 (with FEA7-3) ¬∑ **Severity: CRITICAL**
¬ß7.7 says the proxy adds "authentication, audit, and failure mapping **only**" and "request/response bytes pass through" ‚Äî authorization was put on the ¬ß7.6 CommandPublisher path and **omitted on the CMM path**, which does not traverse the bus so the ¬ß6.8.2 per-topic ACL never covers it. ¬ß6.8.3 defines one mTLS namespace with no cert‚Üíoperation binding. Verified real surface (`CmmReceiverApiRequetsHandler.cs`): `UploadBinCodesToSecsIIWaferMap` ‚Üí **writes the SECS-II wafer map the host reads** (yield/bin falsification), `Alert`/`ExportMapConfirmation` ‚Üí **operator modal/alarm injection**, `DoPostScanMapMatch ‚Üí LoadJobSetup(attacker path)`. This is *new* authenticated external reach to a surface that is loopback-only today.
**Resolution (decided):** (1) the CMM proxy sits behind the same **default-deny per-identity operation allowlist** as ¬ß7.6 (CMM ops are a distinct grant from MES commands); (2) **scope certs to an operation class** (CMM certs ‚âÝ MES certs); (3) individually authorize+audit `UploadBinCodesToSecsIIWaferMap`/ExportMap/setup-path ops; (4) an operation-id allowlist at the proxy so "no translation" ‚âÝ "no gate." "CMM proxy refuses un-allowlisted operations" is a Wave-2 exit criterion with a test. ‚Üí 07 ¬ß7.7, ¬ß7.6, 06 ¬ß6.8.3.

### X7-7 ‚Äî The GEM transition ring (the sole R-6 closure mechanism) has no mechanism: no registered topic, no producer, no storage, no server
**From:** GS7-1 (with GS7-3, GS7-7) ¬∑ **Severity: CRITICAL**
R-6's "no per-site host sign-off required" rests entirely on the ring, which exists only as prose (01:333) and a diagram box. Verified: not a registered topic (snippet 01 defines 11, none is the ring; View 3's "+2" are other topics); not producible by the broker (class-B is a single last-value slot, not last-N); not served as R-R (no ACL'd topic, snippet 09 has no buffer/`Serve`, snippet 13 has no `TransitionRing`). The R-8 fan-out worker produces `(prev,new,seq)` records ‚Äî exactly ring content ‚Äî but no element stores or transports them to the shim. ‚Üí after a reconnect blip during an `Engineering‚ÜíE2P‚ÜíEngineering` failure cycle the host misses the E30 CEIDs = the DI-8 scenario R-6 called closed.
**Resolution (decided):** specify the ring as a first-class contract ‚Äî an **R-R topic `tool.state.replay` served by the ToolManager shim from a bounded last-N buffer the R-8 fan-out worker appends to** (registry entry + ACL + payload `ToolStatePayload[]` with prev/seq); register it, count it in View 3, add it to snippets 09/13, add a TestKit assertion (reconnect-with-gap ‚Üí replayed CEIDs). The same REQ is the natural carrier for the R-6 "completed handshake" (GS7-7). ‚Üí 01 ¬ß1.3.4/View 3, 06 ¬ß6.6/registry, snippets 01/09/13.

### X7-8 ‚Äî The GEM async-accept path assumes a deferred-reply primitive the untouched Cimetrix driver does not have; the only real primitive changes host-visible bytes
**From:** GS7-2 ¬∑ **Severity: CRITICAL**
`GemCommandDisposition.Pending` + `IGemTransaction.CompleteHcack(accepted)` promise "accepted, completed async within the E30 window." Verified: the real host-command contract is **synchronous in-out** ‚Äî `IE30CommandCB.CommandCalled([in]IRemoteCommand, [in,out]eCommandResults)` (`E30.h:62`, `RemoteControl.cs:45`); HCACK is the value written *before the callback returns*; there is no transaction handle. The only async primitive is **`eCmdPerformLater = 4`** (`SecsGemDriverEnums.h:86`, shipped in `RemoteControlToshiba.cs:168`) = **HCACK=4 immediately, true outcome later as a collection event** ‚Äî a *different wire sequence* than today's HCACK-0-on-completion. That contradicts ¬ß1.3.4's "byte-identical" and the P4 "host-visible behavior identical" gate.
**Resolution (decided):** map the accept path to **`eCmdPerformLater` (HCACK=4) + a named completion CEID**; rewrite `CompleteHcack` as "raises the completion event"; **retract the "byte-identical" claim for host *commands*** (it holds for events/state) and move the accept-path change into the P4/P5 host-requalification budget. The *denial* path (reader-thread HCACK denial) is unaffected. ‚Üí 01 ¬ß1.3.4, 03 P4 gate (¬ß3 lane D), 04 ¬ß4.2 P4 row, snippet 13.

### X7-9 ‚Äî Journal compaction regresses the publisher seq high-water below the gateway's independently-persisted high-water ‚Üí fresh wafers silently dropped as duplicates on the routine daily restart
**From:** DI7-1 ¬∑ **Severity: CRITICAL**
R-2's failure mode reborn through the **steady-state healthy path**, evading R-2's fix (epoch fires only on journal *re-creation*, which compaction is not). Sequence: publisher at seq=1000; top seqs get E2E-acked ‚Üí tombstoned ‚Üí compaction (`ReplaceFile`) removes them; highest surviving entry = 998. Clean daily restart ‚Üí `_nextSeq` restored as `max(journal seq)` = **998**, epoch unchanged (¬ß6.3: clean restart keeps epoch). The gateway's per-source high-water lives in the WAL store (07 ¬ß7.5) and does **not** regress ‚Äî still **1000**. Fresh wafers publish seq 999‚Äì1000 ‚Üí `DedupIndex` drops them as duplicates ‚Üí DELIVER_ACK'd ‚Üí tombstoned = **silent loss** until seq climbs past 1000. Extreme case (all entries acked+compacted overnight ‚Üí journal empties ‚Üí high-water restores to 0, while the shared-with-telemetry gateway high-water is thousands) = **a full day of production silently discarded**. No counter records a dedup-drop. Falsifies SYS-1 and TestKit 5. The design *chose* this mechanism (decision-brief R-2 rejected "persist `_nextSeq`" in favor of "restore from journal high-water").
**Resolution (decided):** persist a **monotonic seq high-water sidecar** next to the epoch file (survives dir-clear via a boot seed); restore `_nextSeq = max(sidecar, journal-max)`; **compaction never lowers it**; epoch-on-re-creation unchanged. Closes both the compaction trigger and the epoch-file-cleared trigger (equal/lower epoch reuse). Add a TestKit 5 crash point: "compact-to-empty ‚Üí restart ‚Üí fresh wafer accepted, not deduped." ‚Üí 06 ¬ß6.5/¬ß6.3, 07 ¬ß7.5, snippet 03.

## MAJOR findings (resolutions decided; grouped)

**Gateway durability & backpressure**
- **M-1 (CC7-4):** `Attempts` is per-entry but poison is per-sink ‚Üí one sink's failures poison the other's first attempt. Fix: per-leg `{state, attempts}`, increment only via the WAL-state actor, "attempt" = a completed send while classified reachable. ‚Üí 07 ¬ß7.3/¬ß7.4.
- **M-2 (CC7-5):** "withhold DELIVER_ACK" has no mechanism; the two na√Øve builds either grow an unbounded continuation chain or stall the pipe reader (starving `tool.state`/`PING` ‚Üí broker disconnects the gateway at quota). Fix (4 sentences in 07 ¬ß7.5/¬ß7.8, 06 ¬ß6.5): withhold ‚âÝ block ‚Äî the handler **returns without acking** (deferred-ack list keyed by deliveryId) or sends explicit subscriber-NACK; **the pump reader must never block on a lane**; the broker bounds unacked in-flight per (topic,subscriber); a below-low-watermark event acks/redelivers the deferred set. - **M-3 (CN7-4):** ambiguous (deadline-after-send) legs make downstream dedup **load-bearing, not optional** ‚Äî routine timeout behavior, yet A-6 calls Fleet `messageId` dedup optional and ¬ß7.1 promises "no unrecorded duplication downstream."
